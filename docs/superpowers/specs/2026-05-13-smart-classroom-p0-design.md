# Smart Classroom P0 Local Development Design

## Goal

Build a runnable local P0 version of the cloud-edge-device smart classroom monitoring system described in `readme.md`.

The first version must demonstrate the complete assignment loop:

```text
sensor simulator -> edge processing -> cloud storage and analysis -> dashboard display
```

The system will run locally during development. Later, the cloud service can be deployed to a cloud server and migrated from SQLite to MySQL or another server-grade database.

## Confirmed Scope

P0 includes:

- Python sensor simulator.
- Edge FastAPI service.
- Sliding-window stream processing on the edge.
- Data validation and filtering on the edge.
- Cloud FastAPI service.
- SQLite storage for local development.
- React + Vite dashboard.
- Real large-model API integration with user-provided credentials.

P0 excludes:

- Redis Stream.
- WebSocket push.
- Login and role-based permission management.
- Docker Compose.
- MySQL.
- Cloud server deployment scripts.

These excluded items remain suitable P1/P2 extensions after the local P0 loop works.

## Local Architecture

The local system runs as four processes:

```text
sensor-simulator
  -> edge-service:8001
  -> cloud-service:8000 + SQLite
  -> frontend:5173
```

### Device Layer

`sensor-simulator` generates classroom environment data and sends it to the edge service.

Responsibilities:

- Simulate one or more classrooms.
- Generate temperature, humidity, CO2, light, and people-count values.
- Support normal and abnormal modes.
- Send data to `POST /api/v1/sensor/data` on the edge service.

The simulator does not run complex analysis. This keeps the device layer lightweight and makes the cloud-edge-device division clear.

### Edge Layer

`edge-service` receives raw sensor data and performs real-time processing.

Responsibilities:

- Validate request payloads with Pydantic.
- Reject invalid values, such as impossible temperature or humidity values.
- Maintain a sliding window per room.
- Compute aggregate metrics such as average temperature, average CO2, average light, and maximum people count.
- Detect abnormal events such as high temperature, high CO2, low light, and crowding.
- Upload aggregate snapshots and abnormal events to the cloud service.

The edge service only uploads useful processed data to the cloud. This demonstrates local stream processing and reduced cloud load.

### Cloud Layer

`cloud-service` stores processed data and exposes APIs for the frontend.

Responsibilities:

- Receive aggregate data from the edge service.
- Receive abnormal events from the edge service.
- Store rooms, devices, sensor snapshots, events, and AI analysis records in SQLite.
- Expose latest, history, room, event, and AI analysis APIs.
- Call a real large-model API when the user requests event analysis.

SQLite is used for local development because it avoids local database setup. The database layer should be structured so it can later be replaced by MySQL for cloud deployment.

### Frontend Layer

`frontend` is a React + Vite dashboard.

Responsibilities:

- Show all classroom status cards on the overview page.
- Show latest metrics and historical charts for a selected classroom.
- Show abnormal event lists.
- Trigger AI analysis for an event.
- Display AI analysis results.
- Provide a settings area where users can enter model connection information.

The first frontend version uses REST polling or manual refresh. WebSocket real-time push is intentionally deferred.

## Data Model

The cloud service stores the following entities.

### rooms

Stores classroom metadata.

Required fields:

- `id`
- `room_id`
- `room_name`
- `building`
- `capacity`
- `created_at`

### devices

Stores simulated device metadata.

Required fields:

- `id`
- `device_id`
- `room_id`
- `device_type`
- `status`
- `created_at`

### sensor_data

Stores edge aggregate snapshots and latest room metrics.

Required fields:

- `id`
- `device_id`
- `room_id`
- `temperature`
- `humidity`
- `co2`
- `light`
- `people_count`
- `source`
- `timestamp`
- `created_at`

### edge_events

Stores abnormal events generated by the edge service.

Required fields:

- `id`
- `event_id`
- `room_id`
- `event_type`
- `level`
- `message`
- `metrics`
- `timestamp`
- `created_at`

`metrics` stores structured JSON text in SQLite.

### ai_analysis

Stores large-model analysis results for events.

Required fields:

- `id`
- `event_id`
- `room_id`
- `summary`
- `impact`
- `suggestions`
- `energy_saving`
- `created_at`

`suggestions` stores structured JSON text in SQLite.

## API Design

### Edge API

```text
POST /api/v1/sensor/data
```

Receives raw sensor data from the simulator.

Example request:

```json
{
  "device_id": "sensor-A101-001",
  "room_id": "A101",
  "temperature": 29.1,
  "humidity": 66.5,
  "co2": 1230,
  "light": 180,
  "people_count": 42,
  "timestamp": "2026-05-13T20:00:00"
}
```

### Cloud Edge-Ingestion API

```text
POST /api/v1/edge/aggregate
POST /api/v1/edge/events
```

`/aggregate` stores processed room snapshots. `/events` stores abnormal events.

### Cloud Query API

```text
GET /api/v1/rooms
GET /api/v1/rooms/{room_id}/latest
GET /api/v1/rooms/{room_id}/history
GET /api/v1/events
POST /api/v1/events/{event_id}/analyze
```

These APIs support the dashboard and AI analysis workflow.

## Edge Processing Rules

### Data Cleaning

The edge service rejects clearly invalid data:

- Temperature lower than `-20` or higher than `60`.
- Humidity lower than `0` or higher than `100`.
- CO2 lower than `300` or higher than `5000`.
- Light lower than `0`.
- People count lower than `0`.

### Sliding Window

The edge service keeps a fixed-size window per room. The default window size is `5` to make demonstrations responsive.

Computed metrics:

- `avg_temperature`
- `avg_humidity`
- `avg_co2`
- `avg_light`
- `max_people_count`

### Event Detection

The edge service generates events from the current window:

- `HIGH_TEMPERATURE`: average temperature is greater than `28`.
- `HIGH_CO2`: average CO2 is greater than `1000`.
- `LOW_LIGHT`: average light is lower than `200` and people count is greater than `0`.
- `CROWD`: maximum people count is greater than `50`.

Event level:

- `info` for low light.
- `warning` for moderate high temperature or high CO2.
- `critical` for serious temperature, CO2, or crowding conditions.

The implementation should avoid generating duplicate events every second for the same room and event type. A simple cooldown window is enough for P0.

## Large-Model Integration

The system uses a real large-model API, but credentials are not hard-coded.

Supported configuration:

- `OPENAI_API_KEY`
- `OPENAI_BASE_URL`
- `OPENAI_MODEL`

The implementation should use an OpenAI-compatible chat-completions API shape so users can configure OpenAI, compatible proxies, or compatible domestic model providers when available.

The frontend settings page lets users enter:

- API key.
- Base URL.
- Model name.

For P0, the frontend may send these values to the cloud analysis endpoint for the current request. The cloud service uses them only for that request and does not store the API key in SQLite. The cloud service also supports `.env` fallback values for local development and later deployment.

The analysis endpoint returns structured content:

```json
{
  "summary": "A101 教室当前环境较闷热，CO2 浓度偏高。",
  "impact": "可能影响学生注意力和舒适度。",
  "suggestions": [
    "开启通风设备",
    "将空调调整至 26 摄氏度",
    "适当提高照明亮度"
  ],
  "energy_saving": "在人离开后自动关闭照明和空调。"
}
```

If the model call fails, the API should return a clear error message without crashing the cloud service.

## Frontend Design

The dashboard uses a compact management layout:

- Left navigation.
- Right content workspace.
- Overview page.
- Classroom detail page.
- Event page.
- AI suggestion page or panel.
- Settings page.

### Overview Page

Shows classroom cards with:

- Room ID.
- Temperature.
- CO2.
- People count.
- Status.
- Latest suggestion summary when available.

### Classroom Detail Page

Shows:

- Latest metrics.
- Temperature chart.
- Humidity chart.
- CO2 chart.
- Light chart.
- People-count chart.
- Recent events.
- Recent AI analysis.

### Event Page

Shows:

- Event time.
- Room.
- Event type.
- Level.
- Message.
- Analysis status.
- Action to generate AI analysis.

### Settings Page

Shows fields for:

- API key.
- Base URL.
- Model name.

The frontend should avoid storing the API key in source code. Browser-local persistence is acceptable for local demonstration if clearly scoped to the user's browser.

## Error Handling

Edge service:

- Reject invalid sensor payloads with HTTP 422 or a clear application error.
- Continue running if cloud upload fails.
- Log failed cloud uploads.

Cloud service:

- Return 404 for missing rooms or events.
- Return a clear error if AI credentials are missing.
- Return a clear error if the large-model provider fails.
- Keep SQLite initialization automatic for local development.

Frontend:

- Show loading states.
- Show empty states when no data exists yet.
- Show API errors near the failed action.
- Keep the dashboard usable even when AI analysis is not configured.

## Testing Plan

Backend tests:

- Sliding-window aggregation.
- Data-cleaning rules.
- Event-detection thresholds.
- Cloud ingestion APIs.
- Room latest/history APIs.
- AI analysis error path when credentials are missing.

Frontend verification:

- Vite app starts successfully.
- Dashboard renders without backend data.
- Dashboard renders room cards and events after simulator data arrives.
- AI settings form can pass model configuration to the analysis request.

Manual end-to-end verification:

1. Start cloud service on port `8000`.
2. Start edge service on port `8001`.
3. Start sensor simulator.
4. Start frontend on port `5173`.
5. Confirm latest data appears for `A101`.
6. Confirm abnormal events appear when simulator mode produces high CO2, high temperature, low light, or crowding.
7. Enter model credentials and generate an AI suggestion for one event.

## Migration Notes

The local P0 system should keep cloud-specific assumptions isolated:

- SQLite database URL should be configurable.
- The AI provider should be configurable through request data or `.env`.
- Edge cloud URL should be configurable.
- Frontend cloud API base URL should be configurable.

When moving to a cloud server, the expected migration path is:

1. Deploy `cloud-service` on the server.
2. Replace SQLite with MySQL or another server database.
3. Configure model API credentials in server environment variables.
4. Point local `edge-service` to the server cloud API URL.
5. Build and deploy the frontend.
